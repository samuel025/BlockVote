## 1. Architecture Overview

```
┌─────────────────┐     REST API      ┌──────────────────┐     JSON-RPC      ┌──────────────────────┐
│                 │ ───────────────▶  │                  │ ──────────────▶  │                      │
│  React Frontend │                   │  Express Backend │                   │  Hardhat Local Node  │
│  (Vite + React) │ ◀───────────────  │  (Node.js)       │ ◀──────────────  │  (Ethereum)          │
│                 │                   │                  │                   │                      │
│  - Dashboard    │                   │  - SQLite DB     │                   │  - Groth16Verifier   │
│  - Enrollment   │                   │  - Poseidon Hash │                   │  - VoterEligibility  │
│  - Vote Casting │                   │  - snarkjs Proof │                   │  - Voting            │
│  - Live Results │                   │  - DID Mappings  │                   │  - SmartAccountFactory│
│  - Admin Panel  │                   │                  │                   │  - VotingPaymaster   │
└─────────────────┘                   └──────────────────┘                   └──────────────────────┘
      :5173                                 :3001                                   :8545
```

### Data Flow Summary

1. **Enrollment**: Student submits matric number → Backend verifies against SQLite → Generates DID (Poseidon hash) + enrollment commitment + enrollment secret → Stores mapping in `did_mappings` table.
2. **Verification**: Student submits matric + election ID → Backend reconstructs private inputs → Generates Groth16 zk-SNARK proof + Solidity calldata.
3. **On-Chain Registration**: Frontend submits proof to `VoterEligibility.verifyAndRegister()` → On-chain Groth16 verification → DID marked eligible.
4. **Vote Casting**: Frontend calls `Voting.castVote(didHash, candidateId)` → Contract checks eligibility + prevents double-voting → Vote recorded immutably.
5. **Self-Tallying**: Anyone can call `Voting.getResults()` to read vote counts directly from the blockchain.

## 4. Step-by-Step Setup Guide

### Phase 0: Install Dependencies

```bash
# 1. Clone and enter the project
cd voting-system

# 2. Install root dependencies (Hardhat, circomlib, snarkjs)
npm install

# 3. Install backend dependencies
cd backend && npm install && cd ..

# 4. Install frontend dependencies
cd frontend && npm install && cd ..
```

### Phase 1: ZKP Trusted Setup (Circuit Compilation)

This compiles the circom circuit, performs the Groth16 trusted setup ceremony, and generates the on-chain Solidity verifier.

```bash
node scripts/setup-zkp.js
```

**What it does (5 steps):**

| Step | Action | Output |
|------|--------|--------|
| 1/5 | Compile `eligibility.circom` with `circom` | `circuits/build/eligibility.r1cs`, `eligibility_js/eligibility.wasm` |
| 2/5 | Powers of Tau ceremony (BN128, 2^12) | `pot12_0000.ptau` → `pot12_0001.ptau` → `pot12_final.ptau` |
| 3/5 | Circuit-specific Phase 2 setup | `eligibility_0000.zkey` → `eligibility_final.zkey` |
| 4/5 | Export Solidity verifier | `contracts/Verifier.sol` (overwrites existing) |
| 5/5 | Generate + verify test proof | `test_proof.json`, `test_public_signals.json` |

**Expected output:**
```
=== Phase 1: ZKP Trusted Setup ===
[1/5] Compiling eligibility.circom...
   ✓ Circuit compiled successfully.
[2/5] Starting Powers of Tau ceremony...
   ✓ Powers of Tau ceremony complete.
[3/5] Generating circuit-specific proving and verifying keys...
   ✓ Proving key (zkey) generated.
   ✓ Verification key exported.
[4/5] Exporting Solidity verifier (Verifier.sol)...
   ✓ Verifier.sol written
[5/5] Generating test proof to validate pipeline...
   ✓ Test proof generated and VERIFIED successfully!
```

**Tables created:**
- `students` — 15 rows (12 ACTIVE, 1 GRADUATED, 1 SUSPENDED, 1 for each edge case)
- `did_mappings` — Empty (populated during enrollment)

```bash
npx hardhat node
```

Keep this running in a **dedicated terminal**. It starts a local Ethereum node on `http://127.0.0.1:8545` with 20 pre-funded accounts (10,000 ETH each). Chain ID: 1337.

## 5. Smart Contracts

### 5.1 Groth16Verifier (`Verifier.sol`)

- **Auto-generated** by `snarkjs` during `setup-zkp.js` — do NOT edit manually.
- Pure assembly implementation for gas efficiency.
- Exposes: `verifyProof(uint[2] _pA, uint[2][2] _pB, uint[2] _pC, uint[3] _pubSignals) → bool`
- The 3 public signals are: `[enrollmentCommitment, nullifierHash, electionId]`
- Verification key constants (alpha, beta, gamma, delta, IC points) are hardcoded.

### 5.2 VoterEligibility (`VoterEligibility.sol`)

**Purpose:** Bridges ZKP verification with voter registration.

**Key state:**
- `enrollmentCommitments` — `mapping(bytes32 => bool)`: Whitelist of valid commitments (set by admin)
- `usedNullifiers` — `mapping(bytes32 => bool)`: Prevents double-registration per election
- `eligibleVoters` — `mapping(uint256 => mapping(bytes32 => bool))`: Per-election DID eligibility

**Admin functions:**
- `createElection(uint256 _electionId)` — Reset eligibility state for a new election
- `addEnrollmentCommitments(bytes32[] _commitments)` — Batch-register valid commitments
- `revokeEnrollmentCommitment(bytes32 _commitment)` — Revoke a student's commitment

**Core function — `verifyAndRegister()`:**
```
1. Validate electionId matches currentElectionId
2. Validate enrollmentCommitment is whitelisted
3. Validate nullifierHash not already used (prevents double-registration)
4. Validate didHash not already eligible
5. Call Groth16Verifier.verifyProof() on-chain
6. Mark nullifier used + DID eligible
7. Emit VoterVerified event
```

### 5.3 Voting (`Voting.sol`)

**Purpose:** Core election logic — candidates, voting, self-tallying.

**Key state (all mapped by electionId):**
- `candidates[electionId][candidateId]` — Struct: `{id, name, party, voteCount, exists}`
- `hasVoted[electionId][didHash]` — Prevents double-voting
- `totalVotesCast[electionId]` — Running total

**Admin functions:**
- `initializeElection(id, startTime, endTime)` — Cross-validates with eligibility contract
- `addCandidate(id, name, party)` — Register a candidate
- `endElection()` — Manually stop voting

**Voting:**
- `castVote(bytes32 _didHash, uint256 _candidateId)` — Protected by `whenElectionActive` + `nonReentrant`
  - Checks: `isEligible()` via eligibility contract, `!hasVoted`, candidate exists
  - Increments `voteCount` and `totalVotesCast`

**Self-tallying (public view functions):**
- `getResults(electionId)` — Returns all candidates with vote counts
- `getWinner(electionId)` — Returns the candidate with the most votes
- `hasVoterVoted(electionId, didHash)` — Check if a DID already voted

### 5.4 SmartAccount (`SmartAccount.sol`)

Two contracts in one file:

**VoterSmartAccount:**
- Minimal ERC-4337 smart account owned by a student's DID
- `initialize(bytes32 _didHash, address _owner)` — One-time setup
- `execute(address target, uint256 value, bytes data)` — Proxy calls (onlyOwner)
- Students don't need private keys or ETH

**SmartAccountFactory:**
- Deterministic deployment via CREATE2 (same DID → same address)
- `getOrCreateAccount(bytes32 _didHash, address _owner)` — Deploy or return existing
- `computeAddress(bytes32 _didHash)` — Predict address without deploying

### 5.5 VotingPaymaster (`VotingPaymaster.sol`)

- Simplified ERC-4337 Paymaster for gasless voting
- University pre-funds the contract; it sponsors gas for voting transactions
- `validatePaymasterUserOp()` — Approves sponsorship for ops targeting the Voting contract
- `recordSponsorship()` — Tracks metrics (`totalGasSponsored`, `totalOperationsSponsored`)
- `deposit()` — Admin deposits ETH to the EntryPoint

### 5.6 MockVerifier (`mocks/MockVerifier.sol`)

- Always returns `true` for any proof — used exclusively in unit tests
- Allows testing the full voting flow without compiling the ZKP circuit

## 6. ZKP Circuit

### 6.1 Circuit: `eligibility.circom`

```
Template: EligibilityProof()
Curve: BN128 (alt_bn128)
Proof System: Groth16
Hash Function: Poseidon
```

**Signals:**

| Signal | Visibility | Description |
|--------|-----------|-------------|
| `matricNumber` | Private | Student's matric number as a field element |
| `departmentId` | Private | Department identifier |
| `enrollmentSecret` | Private | Secret issued during enrollment |
| `enrollmentCommitment` | Public | `Poseidon(matricNumber, departmentId, enrollmentSecret)` |
| `nullifierHash` | Public | `Poseidon(matricNumber, electionId)` |
| `electionId` | Public | Current election identifier |

**Constraints:**
1. `Poseidon(matricNumber, departmentId, enrollmentSecret) === enrollmentCommitment`
2. `Poseidon(matricNumber, electionId) === nullifierHash`

**What the proof demonstrates:**
- The prover knows valid enrollment data matching a public commitment
- The nullifier is correctly derived (prevents double-registration)
- Zero personal data is revealed on-chain

### 6.2 Trusted Setup Ceremony

The `setup-zkp.js` script performs a simplified ceremony:

1. **Phase 1 (Universal):** Powers of Tau with 2^12 = 4096 constraint capacity
   - `newAccumulator()` → `contribute()` → `preparePhase2()`
2. **Phase 2 (Circuit-specific):** Generate proving/verifying keys
   - `newZKey()` → `contribute()` → final `.zkey`
3. **Export:** Verification key (JSON) + Solidity verifier contract

> **Note:** In production, the ceremony requires multiple independent contributors. The prototype uses single-contributor entropy for simplicity.

### 6.3 Proof Generation (Backend)

The backend generates proofs in `POST /api/verify`:

```javascript
// 1. Convert matric to field element via keccak256
const matricField = BigInt(keccak256(toUtf8Bytes(matricNumber)).slice(0, 64));

// 2. Compute commitments with Poseidon
const enrollmentCommitment = Poseidon(matricField, departmentId, enrollmentSecret);
const nullifierHash = Poseidon(matricField, electionId);

// 3. Generate Groth16 proof
const { proof, publicSignals } = await snarkjs.groth16.fullProve(input, WASM_PATH, ZKEY_PATH);

// 4. Export Solidity calldata
const calldata = await snarkjs.groth16.exportSolidityCallData(proof, publicSignals);
```

## 10. Trust Model & Security Analysis

### 10.1 What the ZKP Protects

| Property | Guaranteed By |
|----------|--------------|
| **Vote privacy** | ZKP — the smart contract never learns which student cast which vote |
| **No double-voting** | Nullifier hash — deterministic per (student, election) pair |
| **Proof soundness** | Groth16 — computationally infeasible to forge a valid proof |
| **Tally integrity** | Blockchain — immutable, publicly auditable vote counts |
| **Self-tallying** | `getResults()` is a public view function anyone can call |

### 10.2 Centralized Trust Point: DID-to-Matric Mapping

The `did_mappings` SQLite table stores `matric_number ↔ did_hash ↔ enrollment_commitment ↔ enrollment_secret`. This gives the backend:

| Capability | Impact |
|------------|--------|
| **De-anonymization** | Can map on-chain nullifiers/commitments back to real students |
| **Proof forgery** | Holds `enrollment_secret` — could generate proofs for any student |
| **Selective disenfranchisement** | Could refuse enrollment or return invalid proofs |

**Mitigation path (production):** Replace server-side secret storage with a hardware token (USB/smart card). The token generates and holds the `enrollmentSecret` locally. The server only stores the `enrollmentCommitment` (the hash), not the preimage.

### 10.3 Smart Contract Security

- **Re-entrancy:** `Voting.castVote()` uses `nonReentrant` modifier
- **Access control:** `onlyAdmin` modifier on all administrative functions
- **Integer overflow:** Solidity 0.8.20 has built-in checked arithmetic
- **Election timing:** `whenElectionActive` modifier checks `block.timestamp` bounds
- **Double-vote:** `hasVoted[electionId][didHash]` mapping
- **Double-registration:** `usedNullifiers[nullifierHash]` mapping

## 11. Troubleshooting

| Problem | Solution |
|---------|----------|
| `circom: command not found` | Install circom binary or set `CIRCOM_PATH` env var |
| `Database not found` | Run `node backend/seed.js` first |
| `Verifier.sol not found` | Run `node scripts/setup-zkp.js` first |
| Backend 500 on `/api/verify` | Ensure ZKP artifacts exist in `circuits/build/` |
| Frontend shows "Backend Offline" | Start backend with `cd backend && npm start` |
| Frontend shows no blockchain data | Ensure `npx hardhat node` is running and contracts are deployed |
| Contract addresses mismatch | Update `frontend/src/config/deployment.js` with values from `deployment.json` |
| `election ID mismatch` error | Ensure `VoterEligibility.createElection()` is called before `Voting.initializeElection()` |
| Tests fail with `already registered` | Each test run needs a fresh Hardhat node (it resets automatically with `hardhat test`) |
